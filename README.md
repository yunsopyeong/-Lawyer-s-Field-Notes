# # Corporate Bankruptcy Overview

## 문서 목적
이 문서는 법인파산·기업파산 주제의 기본 설명을 위한 마스터 문서이다.  
블로그, 브런치, 프리미엄콘텐츠, 유튜브 대본, FAQ, 상담형 답변 작성 시 공통 기준으로 사용한다.

---

## 1. 기본 정의

법인파산은 법인의 재산을 환가하고, 채권자에게 배당한 뒤, 법인을 소멸시키는 청산형 절차이다.  
쉽게 말하면, 회사를 계속 살리는 방향이 아니라 법인을 정리하는 방향의 제도다.

기업파산이라는 표현도 실무상 많이 쓰이지만, 법률적으로는 법인파산이나 파산절차라는 표현으로 설명하는 것이 더 정확하다.

---

## 2. 법인파산을 검토하는 전형적 상황

아래와 같은 경우 법인파산을 검토하는 일이 많다.

- 급격한 매출 감소로 정상 영업이 어려운 경우
- 금융기관 채무를 연체하고 있는 경우
- 거래처 독촉과 압류 위험이 커진 경우
- 임금 또는 퇴직금 체불 문제가 있는 경우
- 세금 체납이 누적된 경우
- 사실상 영업이 중단된 경우
- 자산보다 부채가 많거나 현금흐름이 완전히 막힌 경우
- 회생보다 청산이 더 현실적인 경우

---

## 3. 법인파산과 폐업은 다르다

실무상 가장 많이 오해하는 부분이 바로 이 지점이다.

폐업은 행정적·세무적 의미의 사업 정리에 가깝다.  
반면 법인파산은 법원의 관여 아래 법인의 재산과 채무를 정리하고, 채권자 배당을 거쳐 법인을 소멸시키는 절차다.

즉, 폐업신고만 했다고 해서 법률적으로 회사 문제가 모두 정리되는 것은 아니다.

### 꼭 함께 설명할 포인트
- 폐업과 법인파산은 같은 말이 아님
- 채권자 대응 문제는 폐업으로 끝나지 않음
- 체불임금, 세금, 연대보증 문제는 별도 검토가 필요할 수 있음

---

## 4. 법인파산과 대표이사 개인책임은 다르다

이 부분도 대표이사들이 매우 자주 묻는 질문이다.

법인파산은 법인 자체의 절차다.  
따라서 대표이사 개인의 채무나 연대보증 문제는 자동으로 함께 정리되지 않는다.

### 대표이사 관련 체크포인트
- 대표이사 연대보증 여부
- 체불임금 관련 책임 문제
- 세금 체납 관련 위험
- 사해행위나 편파변제로 오해받을 수 있는 처분
- 회계자료, 통장자료, 거래내역 정리 여부

### 설명할 때 주의할 점
- 대표이사 개인문제를 법인 절차와 분리해 설명하되
- 실무상 함께 점검해야 할 위험 요소는 반드시 짚어 준다

---

## 5. 법인파산 신청 전 준비자료

법인파산에서는 자료 정리가 매우 중요하다.  
실무에서는 요건 설명보다 자료 정리 수준이 절차 진행에 더 큰 영향을 줄 때도 있다.

### 기본적으로 정리할 자료
- 법인 등기부
- 사업자등록 관련 자료
- 최근 재무자료
- 회계장부
- 회사 통장 거래내역
- 자산 목록
- 채권·채무 목록
- 금융기관 채무 현황
- 세금 체납 현황
- 직원 현황
- 임금·퇴직금 체불 자료
- 거래처 미수금 및 미지급금 현황
- 대표이사 연대보증 여부 관련 자료

### 실무상 자주 문제 되는 부분
- 자료 누락
- 통장 사용 내역 불명확
- 특정 채권자만 먼저 변제한 정황
- 명의 이전 또는 무자료 자산 처분
- 회계자료 부실

---

## 6. 법인파산과 기업회생의 차이

법인파산은 회사를 정리하는 절차이고, 기업회생은 회사를 살려 볼 수 있는지를 검토하는 절차다.

### 파산이 더 가까운 경우
- 영업 지속 가능성이 낮은 경우
- 자금흐름이 사실상 끊긴 경우
- 거래처 신용 회복 가능성이 낮은 경우
- 회생계획안 수행 가능성이 약한 경우

### 회생을 먼저 검토할 수 있는 경우
- 영업 자체는 가능하나 채무 구조조정이 필요한 경우
- 거래처와 매출 기반이 남아 있는 경우
- 계속기업가치가 청산가치보다 높을 가능성이 있는 경우

---

## 7. 대표이사가 자주 묻는 질문

- 회사 문 닫으면 끝나는 것 아닌가요
- 폐업신고만 하면 되는 것 아닌가요
- 법인파산을 하면 대표이사 개인 빚도 같이 없어지나요
- 연대보증은 어떻게 되나요
- 직원 임금과 퇴직금은 어떻게 되나요
- 세금이 남아 있어도 법인파산이 가능한가요
- 법인파산 기간은 얼마나 걸리나요
- 법인파산 비용은 어떤 항목이 있나요

이 질문들은 FAQ, 도입부, 유튜브 오프닝에 반복적으로 활용할 수 있다.

---

## 8. 블로그 작성 시 강조 포인트

- 제목은 질문형 또는 판단형으로 잡는다.
- 첫 문단은 실제 상담형 질문으로 시작한다.
- “폐업과 파산의 차이”를 초반에 반드시 넣는다.
- 중간에 “대표이사 개인책임은 별도 문제”를 별도 소제목으로 분리한다.
- 결론은 “신청 전 준비자료 정리”로 마무리한다.

예시 제목:
- 법인파산을 해야 하는 상황일까요
- 폐업만 하면 끝날까요, 법인파산이 필요한 회사는 따로 있습니다
- 대표이사가 가장 많이 오해하는 법인파산 3가지

---

## 9. 유튜브 대본 작성 시 강조 포인트

- 도입 15초 안에 “폐업과 법인파산은 다릅니다”를 던진다.
- 대표이사가 가장 많이 오해하는 3가지를 먼저 설명한다.
- 절차 설명보다 판단 기준을 먼저 제시한다.
- 공포를 조장하지 않고 차분하게 설명한다.
- 마지막은 “무엇을 먼저 정리해야 하는가”로 끝낸다.

---

## 10. 실무상 자주 오해하는 부분

1. 폐업과 법인파산은 같지 않다  
2. 법인파산을 해도 대표이사 개인문제는 별도로 남을 수 있다  
3. 자료가 정리되지 않으면 절차가 지연될 수 있다  
4. 특정 채권자만 먼저 변제하면 문제가 될 수 있다  
5. 세금과 체불임금 문제는 자동으로 정리되지 않는다  
6. 겁을 먹고 아무 조치를 하지 않는 것이 오히려 더 불리할 수 있다

# 윤소평 변호사 GitHub 업데이트 체크리스트

이 문서는 윤소평 변호사 GitHub 저장소에서 콘텐츠, 프롬프트, 템플릿, README, 운영 문서를 업데이트할 때 지켜야 할 최소 기준을 정리한 체크리스트입니다.  
목적은 단순 업로드가 아니라, 저장소 구조를 안정적으로 유지하고, 민감정보 유출과 운영 혼선을 막는 데 있습니다.

---

## 1. 업데이트 전 확인

- [ ] 이번 수정 주제가 한 가지로 정리되어 있는가  
- [ ] 기존 파일을 보강하는 것이 맞는지, 새 파일을 만드는 것이 맞는지 구분했는가  
- [ ] 파일을 올릴 위치가 현재 폴더 구조와 맞는가  
- [ ] 공개 저장소에 올려도 되는 문서인지 다시 확인했는가  
- [ ] 의뢰인 정보, 사건번호, 연락처, 주민등록번호, 비공개 상담내용, 비공개 계약자료가 포함되어 있지 않은가  
- [ ] API 키, 비밀번호, 토큰, 쿠키, 인증파일 등이 포함되어 있지 않은가  
- [ ] 초안, 메모, 원본 자료와 공개용 정리본을 구분했는가  

---

## 2. 브랜치 원칙

- [ ] `main` 또는 기본 브랜치에 바로 수정하지 않았는가  
- [ ] 작업용 브랜치를 새로 만들어 수정했는가  
- [ ] 브랜치명은 수정 목적이 드러나게 만들었는가  

### 권장 브랜치명 예시
- `docs/update-readme`
- `prompts/personal-rehabilitation-rewrite`
- `content/corporate-bankruptcy-faq`
- `ops/github-checklist-update`

### 브랜치명 작성 원칙
- 너무 짧게 쓰지 않는다
- 무엇을 바꾸는지 한눈에 보이게 쓴다
- 여러 주제를 한 브랜치에 섞지 않는다

---

## 3. 파일 수정 원칙

- [ ] 한 번의 업데이트에는 한 주제만 반영했는가  
- [ ] README, 프롬프트, 템플릿, 자료맵을 한 PR에 무리하게 섞지 않았는가  
- [ ] 기존 제목, 폴더 체계, 번호 체계를 함부로 바꾸지 않았는가  
- [ ] 같은 주제의 중복 파일을 또 만들고 있지 않은가  
- [ ] 기존 문서와 충돌하는 오래된 표현이 남아 있지 않은가  

### 특히 주의할 점
- 같은 주제를 이름만 바꿔 여러 파일로 늘리지 않는다
- 기존 마스터 문서가 있으면 먼저 그 문서를 업데이트할 수 있는지 본다
- 파일명은 가능하면 일관되게 유지한다

---

## 4. 민감정보 점검

아래 내용은 GitHub에 올리지 않는 것을 원칙으로 합니다.

- [ ] 의뢰인 실명
- [ ] 주민등록번호, 주소, 연락처
- [ ] 사건번호 전문
- [ ] 상담 녹취, 상담 메모 원문
- [ ] 계약서 원본 PDF
- [ ] 신분증, 등본, 가족관계증명서 등 원본 자료
- [ ] 계정 비밀번호
- [ ] API 키, 토큰, 인증정보
- [ ] 비공개 내부 운영자료 원문

### 업로드 가능한 방향
- 익명화한 예시
- 공개 가능한 템플릿
- 일반화한 체크리스트
- 발행용 프롬프트
- README, 운영 규칙, 문체 기준, 품질검수 문서

## 5. 커밋 메시지 원칙

커밋 메시지는 나중에 봐도 무엇을 바꿨는지 바로 이해되어야 합니다.

### 금지 예시
- update
- fix
- 수정
- 변경

### 권장 예시
- `개인회생 프롬프트 FAQ 5개 추가`
- `법인파산 개요 문서 표현 정리`
- `README GitHub 운영 규칙 보강`
- `최종검수 체크리스트 금지표현 수정`

### 체크
- [ ] 무엇을 바꿨는지 보이는가  
- [ ] 왜 바꿨는지 추측 가능한가  
- [ ] 너무 포괄적이지 않은가  

---

## 6. Pull Request 작성 전 확인

- [ ] base 브랜치가 맞는가  
- [ ] compare 또는 head 브랜치가 맞는가  
- [ ] 변경 파일 수가 너무 많지 않은가  
- [ ] 이번 PR의 목적을 한 문장으로 설명할 수 있는가  
- [ ] 불필요한 파일까지 같이 들어가지 않았는가  
- [ ] `.gitignore`에 넣어야 할 파일이 커밋되지 않았는가  

### PR 제목 예시
- `개인회생 프롬프트 문구 보강`
- `법인파산 README 구조 정리`
- `GitHub 운영 체크리스트 추가`

### PR 본문 예시
- 무엇을 수정했는가:
- 왜 수정했는가:
- 영향받는 파일:
- 확인할 포인트:
- 민감정보 포함 여부: 없음

---

## 7. 업데이트 후 확인

- [ ] PR 화면에서 변경 범위가 의도한 내용과 같은가  
- [ ] 오탈자, 파일명, 제목, 링크가 깨지지 않았는가  
- [ ] README 또는 내부 문서의 링크가 끊기지 않았는가  
- [ ] 기존 운영 원칙과 충돌하는 문장이 없는가  
- [ ] 중복 파일을 만든 경우 통합 필요성이 없는가  
- [ ] merge 후 정리할 브랜치가 무엇인지 확인했는가  

---

## 8. 절대 피할 것

- [ ] `main`에 직접 수정
- [ ] 여러 주제를 한 번에 섞어서 올리기
- [ ] 민감정보 포함 파일 업로드
- [ ] 파일명 체계 무너뜨리기
- [ ] 같은 주제 중복 파일 양산
- [ ] “update” 같은 의미 없는 커밋 메시지
- [ ] base 브랜치 확인 없이 PR 생성
- [ ] 검토 없이 바로 merge

## 9. 윤소평 변호사 저장소 기준 실무 원칙

이 저장소는 일반 개발 저장소라기보다  
법률콘텐츠, 프롬프트, 템플릿, 운영 기준을 관리하는 저장소에 가깝습니다.  
따라서 아래 원칙을 우선합니다.

1. 기본 브랜치는 완성본만 둔다  
2. 작업 브랜치는 실험본과 수정본만 둔다  
3. GitHub에는 공개 가능한 정리본만 올린다  
4. 사건자료 원본보다 익명화된 기준서와 템플릿을 올린다  
5. 중복 문서를 늘리기보다 기존 마스터 문서를 보강한다  
6. 업데이트할수록 구조가 더 단순해져야 한다
