# README GitHub 운영문구 반자동 반영안

이 문서는 저장소 `README.md`에 바로 붙여넣기 위한 운영문구 초안입니다.  
현재 저장소는 법률 콘텐츠 기준서와 프롬프트를 누적하는 용도이므로, `main` 직접 수정이 아니라 **작업 브랜치 → PR → 검토 → merge** 흐름으로 운영하는 것을 기본 원칙으로 합니다.

---

## GitHub 운영 방식

이 저장소는 단순 문서 보관함이 아니라,  
윤소평 변호사 법률 콘텐츠 시스템을 안정적으로 누적하기 위한 운영 저장소입니다.

그래서 문서 추가나 수정은 아래 원칙을 따릅니다.

- `main`에는 최종 확정본만 둡니다.
- 작업은 항상 별도 브랜치에서 진행합니다.
- 수정 내용은 PR로 검토한 뒤 반영합니다.
- README, 프롬프트, 템플릿, 운영 문서는 한 번에 너무 많이 섞지 않습니다.
- 민감정보, 사건 원본자료, 실명 자료는 GitHub에 올리지 않습니다.

---

## 프로젝트 결과물 반자동 반영 흐름

이 저장소는 ChatGPT 프로젝트에서 만든 결과물을 곧바로 `main`에 넣지 않습니다.  
대신 아래 흐름으로 반자동 반영하는 것을 권장합니다.

1. ChatGPT 프로젝트에서 최종 원고 또는 운영문서 초안 작성  
2. 지정한 작업 브랜치에 결과물 저장  
3. draft PR 자동 또는 수동 생성  
4. 제목, 본문, 파일 경로, 민감정보 포함 여부 검토  
5. 검토 후 `main` 또는 상위 작업 브랜치로 merge

이 방식의 핵심은 **자동화보다 검토 가능성**입니다.  
법률 콘텐츠와 운영 문서는 잘 쌓는 것보다, 잘 검토하며 쌓는 것이 더 중요하기 때문입니다.

---

## 브랜치 운영 원칙

브랜치명은 무엇을 바꾸는지 바로 보이도록 만드는 편이 좋습니다.

예시:

- `docs-readme-refresh`
- `ops-issue-template-update`
- `prompts-personal-rehabilitation-update`
- `topics-corporate-bankruptcy-faq`
- `quality-final-review-checklist`

브랜치는 아래처럼 구분해 두는 것이 좋습니다.

- `main`: 최종본
- `github`: GitHub 운영 관련 중간 통합 브랜치
- 주제별 브랜치: 개별 작업용
- 운영 브랜치: README, 템플릿, 체크리스트 수정용

---

## PR 운영 원칙

PR은 단순 병합 요청이 아니라,  
무엇을 왜 바꿨는지 남기는 검토 문서입니다.

그래서 PR에는 아래 항목이 들어가는 것이 좋습니다.

- 무엇을 수정했는가
- 왜 수정했는가
- 영향받는 파일은 무엇인가
- 리뷰할 때 중점적으로 볼 부분은 무엇인가
- 민감정보 포함 여부
- 셀프 체크리스트

법률 콘텐츠 저장소에서는 특히 아래를 꼭 확인해야 합니다.

- 과장광고 문구가 들어가 있지 않은가
- 기존 문체 기준과 충돌하지 않는가
- 법률 근거 없는 일반론만 남아 있지 않은가
- 공개 저장소 기준으로 올려도 되는 내용인가

## 업로드 금지 자료

이 저장소에는 아래 자료를 올리지 않는 것을 원칙으로 합니다.

- 의뢰인 실명
- 주민등록번호, 주소, 연락처
- 사건번호 전문
- 상담 녹취 원문
- 계약서 원본 PDF
- 비공개 상담자료
- 비밀번호, 토큰, API 키
- 증거자료 원본 파일

GitHub에는 공개 가능한 템플릿, 기준서, 프롬프트, 운영문서, 익명화된 예시만 올립니다.

---

## 권장 작업 순서

실제 운영은 아래 순서가 가장 안정적입니다.

1. 프로젝트에서 초안 작성  
2. 기존 문서와 중복 여부 확인  
3. 새 브랜치 생성  
4. 관련 파일만 업로드  
5. PR 생성  
6. 셀프 체크  
7. 검토 후 merge

한 번에 많은 파일을 밀어 넣기보다,  
한 주제씩 끊어서 반영하는 편이 장기적으로 훨씬 관리가 쉽습니다.

**빠르게 올리는 것보다, 검토 가능한 구조로 정확하게 쌓는 것**을 목표로 합니다.
